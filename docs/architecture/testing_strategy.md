# Testing Strategy

## Purpose

Define the test taxonomy: which tests exist, where they live, what they check, and what shapes are
forbidden.

## Owns

- The three test categories: `tests/cases`, `tests/ir`, `tests/unit`.
- The contract of each category: scope, input form, expected output form.
- The rule that ad-hoc C++ semantic tests are forbidden.

## Does Not Own

- Individual test contents.
- Test runner implementation details.
- Benchmark methodology (covered elsewhere).

## Core Invariants

1. Any test that validates SystemVerilog semantics must live in `tests/cases`. There is no partial
   exemption for "small", "minimal", or "helper" cases; all SystemVerilog-semantic coverage is case
   coverage.
2. `tests/cases` holds SystemVerilog project-style tests. Each case is a folder with its own sources
   and a `test.yml` declaring expected behavior. Cases run end-to-end through the compiler and
   simulate.
3. `tests/ir` holds IR-level golden tests for HIR, MIR, LIR, and projections. Each test pins the
   canonical form of a specific construct at a specific layer. IR tests pin shape; they do not
   validate SystemVerilog semantic correctness.
4. `tests/unit` holds C++ unit tests for internal utilities that have no SystemVerilog-semantic
   dimension. Unit tests must not assert on HIR, MIR, or LIR shape to prove language correctness.
5. Every SystemVerilog semantic behavior is covered by exactly one owning case test path. Coverage
   lives in `tests/cases`, never duplicated in unit tests.
6. Golden outputs under `tests/ir` are regenerated by a deterministic command. Hand-editing goldens
   is a review red flag.

## Boundary to Adjacent Layers

- Case tests exercise the full pipeline. A failure in a case test may point at any layer.
- IR tests bisect a failure to a specific layer. An IR golden regression narrows the suspect to the
  producer of that layer.
- Unit tests isolate utility code with no cross-layer dependencies.

## Forbidden Shapes

- A C++ test that constructs compiler objects and asserts on their fields to check SystemVerilog
  semantics. Such coverage belongs in case or IR tests.
- A case test without a `test.yml` (or equivalent manifest).
- An IR test whose golden is produced by a non-deterministic pass.
- A unit test that depends on slang, HIR, MIR, or LIR shape.
- A unit test that asserts on HIR, MIR, or LIR shape to prove a SystemVerilog-language behavior.
  Language behavior coverage is case coverage.
- Using labels like "partial semantics", "small behavior", or "minimal repro" to justify placing
  SystemVerilog-semantic coverage outside `tests/cases`.
- Tests that exercise code outside the current source tree.
- Relying on file-path regex as the primary classification between test categories. Category
  membership is established by directory layout, not by matching file names.

## Notes / Examples

A case folder typically contains `top.sv`, optional support sources, and `test.yml`. The manifest
declares expected outputs, simulation length, and any flags. Adding a new SystemVerilog feature
means adding a case under `tests/cases` first, then extending `tests/ir` goldens if the feature
changes canonical IR shape.
